# 真实数学建模比赛工作流调研

调研日期：2026-06-09

本文基于公开的真实项目仓库、竞赛规则、经验总结和优秀论文资源，分析参赛用户实际如何完成一场数学建模比赛，以及 `ModelingPaperKit` 的 Skill / Agent / 全栈产品应该真正解决什么问题。

## 1. 关键结论

真实用户最需要的不是“全自动建模”，而是一个能在高压比赛周期中减少失误、保持进度、沉淀交付物的工作系统。

真实比赛中最常见的痛点是：

1. **时间极短**：CUMCM 通常 3 天左右，MCM/ICM 约 4 天多。用户要同时完成读题、选题、建模、编程、结果分析、论文写作和提交。
2. **分工混乱**：一般 3 人组队，常见分工是论文、编程、建模/资料，但中后期会交叉混乱。
3. **模型现学现用**：真实参赛经验常提到，不可能赛前精通所有模型，关键是知道“什么问题用什么模型、输入输出是什么”。
4. **论文质量决定上限**：真实项目复盘中经常提到，图表不统一、排版错误、摘要打磨不足会直接影响结果。
5. **合规风险高**：CUMCM 和 MCM/ICM 都有严格匿名、引用、AI 使用、提交文件要求。
6. **最后 6 小时最危险**：容易出现 PDF 未生成、图片缺失、引用未解析、文件名错误、身份信息泄露、摘要没覆盖结果等提交事故。

因此产品应优先帮助用户：

- 管理比赛进度。
- 管理论文结构。
- 组织代码、数据、图表和结果。
- 自动编译和排错。
- 做提交前合规检查。
- 记录 AI 使用和引用来源。
- 保证最终 PDF 和支撑材料一致。

## 2. 真实项目观察

### 2.1 公开竞赛项目通常包含什么

真实 CUMCM / MCM 项目仓库通常包含：

- 论文 PDF。
- 源代码。
- 数据处理脚本。
- 图表或中间结果。
- README 说明。
- 有些项目还包含答辩 PPT 或支撑材料。

例如，`kalipolis/CUMCM2023_C` 明确说明其仓库包含 2023 国赛 C 题的论文、代码和数据处理，并在复盘中提到模型创新不足、图表格式不统一、论文有少量排版错误、论文打磨次数不够。这个复盘非常说明问题：用户真正痛的是 **做出了结果之后，论文交付质量没兜住**。

`ydchen0806/24ICM_E_O_Award_Paper_code` 是 2024 MCM/ICM E 题 Outstanding Winner 冠名奖项目，仓库内容包括完整参赛论文和模型分析源代码，并说明使用 MGWR 软件进行分析。这类高质量项目也说明：最终产物不是一个“模型”，而是 **论文 + 源代码 + 方法说明 + 可复现分析链**。

### 2.2 资源库反映的用户准备方式

大型资源库如 `zhanwen/MathModel` 收集了优秀论文、赛题、算法、模板、教材、PPT、Matlab 教程等。这说明用户赛前并不是只找一个工具，而是在准备：

- 常见算法。
- 历年优秀论文。
- 论文模板。
- 编程环境。
- 竞赛规则。
- 资料检索路径。

这对本项目的启发是：`ModelingPaperKit` 不需要自己变成资源库，但应提供清楚入口，告诉用户哪些资料应该作为 references，哪些资料不应在比赛期间违规使用。

## 3. 真实参赛时间线

结合公开经验总结和比赛规则，一个典型流程如下。

### 赛前 1-4 周

用户真实任务：

- 确定队友和分工。
- 熟悉 Python / Matlab / LaTeX。
- 准备论文模板。
- 做几次模拟训练。
- 建立常用模型的“输入-输出-适用场景”索引。
- 准备画图和表格模板。

产品应支持：

- 一键检查本地 TeX 环境。
- 一键编译示例论文。
- 生成比赛项目结构。
- 提供常见章节骨架。
- 提供常见模型结果如何写入论文的片段。

### 第一天：读题、选题、拆题

用户真实任务：

- 阅读所有题目。
- 判断每题数据量、难度、建模方向。
- 快速检索背景知识和可用方法。
- 确定题目。
- 拆成子问题。
- 给出初版技术路线。

产品应支持：

- 题目拆解模板。
- 子问题矩阵：目标、输入、输出、约束、候选模型。
- 风险评估：数据难度、模型难度、论文表达难度。
- 初始化论文结构。

注意：

- CUMCM 竞赛期间有严格交流和平台浏览限制，工具必须避免引导用户到公开平台讨论赛题。
- MCM/ICM 允许使用 inanimate sources，但必须引用，不能向队外人员寻求帮助。

### 第二天：建模、编程、初稿并行

用户真实任务：

- 数据清洗。
- 模型实现。
- 生成初步结果。
- 同步写问题重述、假设、符号说明、模型建立。
- 生成第一批图表。

产品应支持：

- 代码、数据、图表目录规范。
- 图表清单和论文引用清单。
- 结果表模板。
- 将模型输出转成论文段落。
- 提醒“有结果就写入论文”，避免最后堆积。

### 第三天：结果强化、论文成型

用户真实任务：

- 完成主要实验。
- 做敏感性分析、误差分析、模型评价。
- 图表统一样式。
- 写摘要初稿。
- 补充参考文献。
- 反复编译论文。

产品应支持：

- build-fix 循环。
- 图表格式检查。
- 引用和 label 检查。
- 摘要覆盖检查。
- 章节完整性检查。

真实经验里，摘要经常需要反复修改。工具应把摘要作为单独高优任务，而不是最后随便生成。

### 最后 6-12 小时：终稿和提交

用户真实任务：

- 删除身份信息。
- 检查承诺书、编号页、Summary Sheet、AI 声明。
- 确认 PDF 大小。
- 确认支撑材料。
- 确认文件命名。
- 最终提交。

产品应支持：

- final review 模式。
- PDF/log/引用/图片/身份信息检查。
- 支撑材料清单。
- AI 使用声明清单。
- 按赛事生成提交 checklist。

这里是 `ModelingPaperKit` 最应该重点发力的阶段。

## 4. CUMCM 的真实约束

基于 2026 年规则和格式规范，需要特别注意：

- 参赛论文若违反章程、格式规范或承诺书要求，会被取消评奖资格。
- 指导教师在竞赛期间不得解释赛题、提供建议、资料、论文修改等。
- 参赛队员不得与队外任何人讨论赛题，也不得在贴吧、QQ 群、微信群、知乎、小红书、CSDN、GitHub 等平台浏览、发布或讨论与赛题相关的内容。
- 可以使用 AI 工具作为辅助，但必须对作品原创性、真实性、准确性负责，并遵守 AI 使用规定。
- 论文电子版应为单独文件，PDF 或 Word，建议 PDF，大小不超过 20MB。
- 电子版第一页必须为摘要专用页，承诺书和编号专用页不要放入电子版论文。
- 支撑材料至少应包含可运行源程序、自主查阅使用的数据资料、较大篇幅中间结果图表等。
- 支撑材料应压缩为 RAR 或 ZIP，大小不超过 20MB，文件中不能出现身份和学校赛区信息。
- 使用 AI 工具时，应在正文相应位置标注，并在参考文献列出工具名称、版本、机构、日期；支撑材料中还需要 AI 工具使用详情说明。

对产品的直接影响：

- CUMCM 模式必须有“竞赛期间安全模式”。
- 不能默认联网搜索赛题相关内容。
- 必须有身份信息扫描。
- 必须有 AI 使用记录导出。
- 必须区分“纸质/完整模板”和“电子提交版”。
- 必须支持支撑材料清单。

## 5. MCM/ICM 的真实约束

基于 COMAP 2026 instructions：

- MCM/ICM 2026 时间为 2026-01-29 17:00 EST 到 2026-02-02 20:00 EST，PDF 提交截止为 21:00 EST。
- 2026 年有 25 页限制，Summary Sheet、正文、参考文献、目录、附录、代码和问题特定要求都计入 25 页。
- 队伍可使用网页、书籍、论文、数据库等 inanimate resources，但必须引用所有来源。
- 不能向队外人员寻求帮助，也不能发布或分享题目、解题过程或部分成果。
- 论文必须是英文、PDF、至少 12pt 可读字体。
- 每页必须只包含团队 Control Number 等允许信息，不能出现学生、导师、学校等身份信息。
- Summary Sheet 是第一页，且评委非常重视摘要质量。
- AI 使用需要在报告中明确说明模型和用途，并在参考文献/inline citation 中体现；AI 使用报告附在 25 页方案之后，不计入 25 页。
- 提交文件名应使用团队 Control Number，文件小于 25MB，不提交额外程序、数据库或其他文件。

对产品的直接影响：

- MCM 模式必须检查 25 页限制。
- 必须检查 Summary Sheet 是否第一页。
- 必须扫描身份信息。
- 必须检查 Control Number。
- 必须有来源引用清单。
- 必须有 AI use report。
- 不应把代码文件打包进 MCM 提交物，但可以保留在本地 workspace。

## 6. 用户真实想怎么做

真实用户不是线性地完成比赛，而是在多个任务间来回切换：

```text
读题 <-> 查资料 <-> 写代码 <-> 出图 <-> 写论文 <-> 改模型 <-> 改论文 <-> 编译 <-> 检查
```

因此产品不应强迫用户走一个僵硬流程。更好的方式是提供几个“可随时调用的模式”：

### 6.1 选题/拆题模式

用户输入赛题或手动录入题目摘要，系统输出：

- 子问题拆解。
- 数据需求。
- 候选模型。
- 难度风险。
- 论文结构建议。

### 6.2 建模记录模式

用户把模型思路、代码输出、图表结果丢给系统，系统负责：

- 记录模型版本。
- 记录参数和数据来源。
- 生成论文中的模型描述草稿。
- 生成图表 caption 和结果解释。

### 6.3 写作推进模式

系统根据章节状态提醒：

- 哪些章节为空。
- 哪些章节缺公式。
- 哪些章节缺结果图表。
- 哪些子问题没有对应结论。

### 6.4 编译排错模式

系统自动：

- 编译。
- 读 log。
- 提取错误。
- 做最小修复。
- 重新编译。

这是最适合自动化的模式。

### 6.5 终稿检查模式

系统只检查不主动改：

- PDF 是否存在。
- 页数/大小是否合规。
- 身份信息。
- AI 使用声明。
- 支撑材料。
- 图表引用。
- 参考文献。
- 特殊页面。

## 7. 对 Skill / Agent 的修正建议

基于真实用户工作流，之前的设计需要强调以下几点：

### 7.1 增加比赛安全模式

新增概念：

```text
contest_safety_mode:
  off: 赛前/赛后普通模式
  cumcm_active: 国赛比赛期间
  mcm_active: 美赛比赛期间
```

在 `cumcm_active` 下：

- 不主动联网搜索赛题相关内容。
- 不打开 GitHub/CSDN/知乎/小红书等可能涉及赛题讨论的平台。
- 只允许读取用户本地文件和官方规则文件。
- 所有 AI 生成内容进入 AI 使用记录。

在 `mcm_active` 下：

- 可以使用公开 inanimate resources，但必须记录来源。
- 不向互动平台提问，不上传题目或解题过程。
- 所有来源进入 reference log。

### 7.2 增加 AI 使用记录器

真实比赛已经明确要求 AI 使用透明。应新增：

```text
.paperkit/ai_usage_log.json
```

记录：

- 工具名称。
- 模型版本。
- 使用时间。
- 使用目的。
- 用户 prompt 摘要。
- 采纳内容。
- 人工修改说明。

后续可导出：

- CUMCM AI 工具使用详情 PDF/Markdown。
- MCM Report on Use of AI。

### 7.3 增加支撑材料管理

CUMCM 特别需要：

```text
.paperkit/supporting_materials.json
supporting/
├── code/
├── data/
├── results/
└── README.md
```

检查：

- 是否包含可运行源程序。
- 是否包含自主查阅数据。
- 是否包含较大中间结果。
- 是否和论文附录清单一致。
- 是否含身份信息。

### 7.4 增加时间线视图

真实用户需要知道“现在该做什么”，尤其比赛后期。

建议 Agent 状态中加入：

```json
"timeline": {
  "phase": "day_2_modeling",
  "deadline": "...",
  "next_milestone": "first_complete_draft",
  "risk": "summary_missing"
}
```

WebUI 可以显示：

- 当前阶段。
- 下一里程碑。
- 剩余风险。
- 最后提交 checklist。

### 7.5 增加结果-论文一致性检查

真实论文常见问题是代码结果和论文表格不一致。

建议维护：

```text
results/
figures/
tables/
.paperkit/result_manifest.json
```

记录每张图/表来源：

- 数据文件。
- 脚本。
- 生成时间。
- 插入章节。
- label。

## 8. 对产品功能优先级的影响

结合真实比赛，优先级应调整为：

### P0：最急缺

- 编译日志摘要。
- 提交前合规检查。
- 身份信息扫描。
- 模板/章节状态检查。
- 图表/引用闭环检查。
- CUMCM/MCM 提交规则 references。
- AI 使用记录。

### P1：强需求

- 支撑材料管理。
- 结果 manifest。
- 摘要专项检查。
- 比赛安全模式。
- CLI Agent 的 build/review/fix-log。

### P2：有价值但后置

- 选题/拆题模式。
- 写作推进模式。
- 时间线视图。
- Web 工作台。
- PDF 预览。

### P3：谨慎推进

- 自动建模。
- 自动写完整论文。
- 多 Agent 自主协作。
- RAG 模型库。
